Authorization Code Flow

How SSO works, step by step

Your client app never handles passwords. It redirects here, we authenticate the user, and send back a short-lived code.

1
Client App
Register your app first
Go to Register to get your client_id and client_secret. Store these securely — client_secret is shown only once.
2
GET
Redirect user to /o/authorize
Send the user here with client_id, redirect_uri, response_type=code, and a random state you generate. If they already have a session, they skip login entirely — that's the SSO magic.
3
Auth Server
User logs in and approves
We show the login + consent page. If the user already has a session from another client, they only see the consent screen (no password again). On approval, we redirect back to your redirect_uri?code=...&state=....
4
POST
Exchange code for tokens at /o/token
Your server (not browser) sends the code + client_secret to /o/token. You get back an access_token (exp 10 min), id_token, and refresh_token (exp 7 days). The code expires in 5 minutes and is one-time use.
5
GET
Fetch user profile from /o/userinfo
Use the access_token as a Bearer token. Returns the user's name, email, and profile fields. Verify the JWT signature using /.well-known/jwks.json — never trust a token you haven't verified.
6
POST
Refresh access token when it expires
Access tokens expire in 7 days. Send grant_type=refresh_token to /o/token with your client_id and client_secret. You get a new access_token without bothering the user.

callback.html — exchange code for tokens 
// Runs when OIDC provider redirects back with ?code=...&state=...
async function exchangeCode() {
  const params  = new URLSearchParams(window.location.search);
  const code    = params.get("code");
  const state   = params.get("state");
  const saved   = sessionStorage.getItem("oauth_state");
 
  // ⚠️ state check skipped in your current setup — add this later
  // if (state !== saved) throw new Error("State mismatch");
  sessionStorage.removeItem("oauth_state");
 
  const response = await fetch("http://localhost:5000/o/token", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      code,
      client_id:     "YOUR_CLIENT_ID",
      client_secret: "YOUR_CLIENT_SECRET", // ⚠️ exposed — see security note
      redirect_uri:  "http://localhost:3000/callback.html",
      grant_type:    "authorization_code"
    })
  });
 
  const data = await response.json();
 
  // access_token + id_token → localStorage (readable by JS)
  if (data.access_token) localStorage.setItem("access_token", data.access_token);
  if (data.id_token)     localStorage.setItem("id_token",     data.id_token);
  // refresh_token → HttpOnly cookie (set by server, JS cannot read it)
 
  window.location.href = "/"; // redirect to home/dashboard
}

middleware/authenticate.ts — protect API routes 
import * as jose from "jose";
 
// Fetches public keys from OIDC provider — cached automatically by jose
const JWKS = jose.createRemoteJWKSet(
  new URL("http://localhost:5000/.well-known/jwks.json")
);
 
export const authenticate = async (req, res, next) => {
  const authHeader = req.headers.authorization;
 
  if (!authHeader || !authHeader.startsWith("Bearer ")) {
    return res.status(401).json({ error: "Unauthorized" });
  }
 
  const token = authHeader.split(" ")[1];
 
  try {
    // Verifies signature, expiry, issuer automatically
    const { payload } = await jose.jwtVerify(token, JWKS, {
      algorithms: ["RS256"],
      issuer:     "http://localhost:5000"
    });
 
    req.user = payload; // { sub, email, name, ... } available in route
    next();
 
  } catch (err) {
    return res.status(401).json({ error: "Invalid or expired token" });
  }
};

refresh.js — get new access token when expired 
// refresh_token is in HttpOnly cookie — browser sends it automatically
// no need to read it from localStorage, credentials:"include" does the job
async function refreshToken() {
 
  const res = await fetch("http://localhost:5000/o/token", {
    method:      "POST",
    credentials: "include", // ✅ sends HttpOnly cookie automatically
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      grant_type:    "refresh_token",
      client_id:     "YOUR_CLIENT_ID",
      client_secret: "YOUR_CLIENT_SECRET"
    })
  });
 
  const data = await res.json();
 
  if (data.access_token) {
    localStorage.setItem("access_token", data.access_token); // ✅ new access token
  } else {
    // refresh token expired — force login
    localStorage.clear();
    window.location.href = "/";
  }
}
⚠️ Security note about this setup
Access tokens are stored in localStorage for demonstration purposes. Refresh tokens are stored in HttpOnly cookies and are never accessible from JavaScript, reducing the risk of token theft through XSS attacks. In production, also move the token exchange to a backend server so client_secret is never exposed in browser source.
Single Sign-On

How SSO works across clients

The Identity Provider maintains a server-side session. Once a user logs in to one client, other clients skip the login screen entirely.

Step 1 — First login (Client A)
User visits Client A → redirected to /o/authorize → enters credentials → session cookie set on auth server → consent shown → code issued → tokens returned. User is now logged in.
Step 2 — SSO login (Client B, same browser)
User visits Client B → redirected to /o/authorize → auth server finds existing session cookie → login page skipped entirely → consent shown for Client B → code issued → tokens returned. No password entered.
✓ Already consented (Client B, returning visit)
Session exists + binding exists → auth server skips both login and consent → code issued instantly → seamless SSO with zero user interaction.
✕ Different browser or incognito
No session cookie present → full login required again. SSO is browser-scoped, not account-scoped.
Tokens

What's inside the tokens

Both access_token and id_token are RS256-signed JWTs. Verify with the public key at /.well-known/jwks.json.

Claim Type Description
sub string User's unique ID — use this as the stable identifier in your DB
email string User's email address
email_verified string "true" or "false" — check before trusting the email
given_name string First name
family_name string Last name
name string Full name (first + last joined)
picture string? Profile image URL if set, otherwise absent
iss string Issuer — must equal http://localhost:5000
aud string Your client_id — always verify this matches yours
exp number Expiry — Unix timestamp, 1 hour from issue
Security

Do's and don'ts

Common mistakes that will break your integration or open security holes.

✓
Do these
  • Generate a fresh random state with crypto.randomUUID() for every login attempt
  • Save state in sessionStorage before redirect — verify it matches on callback
  • Always attach Authorization: Bearer <token> header on every API request
  • Verify JWT using jose.jwtVerify() with JWKS on your API server middleware
  • Use sub (not email) as stable user identifier in your DB
  • Handle 401 responses by calling refreshToken() then retrying
  • Clear all localStorage tokens on logout
  • Ensure redirect_uri exactly matches what you registered
✕
Never do these
  • Never skip state validation on callback — it prevents CSRF attacks
  • Never trust a JWT without verifying its signature via JWKS
  • Never reuse an auth code — one-time use, expires in 5 min
  • Never use email as a primary key — users can change it, use sub
  • Never send requests to protected routes without the Bearer token
  • Never store sensitive user data from JWT in localStorage long-term
  • Never call /o/token without client_secret — request will be rejected
  • Never ignore 401 errors — always refresh or redirect to login
